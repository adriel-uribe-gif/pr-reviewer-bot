Under the Hood

How Bob connects to GitHub

Bob doesn't come with GitHub built in. We built a bridge. It has 3 pieces.

🔧 The Bridge (MCP Server)

A small program that sits between Bob and GitHub. Think of it like a translator — Bob speaks "AI", GitHub speaks "API", and our bridge translates. It gives Bob 6 new abilities it didn't have before.

🔑 The Key Card (GitHub Token)

A secret code that proves to GitHub you're authorized. When the bridge shows this token, GitHub says "OK, you're allowed in." Without it, the bridge gets blocked at the door.

📋 The Contact List (mcp.json)

A settings file that tells Bob the bridge exists and how to start it. Without this file, Bob wouldn't know the bridge was there — like having a phone with no contacts saved.

Why Not Just Use Copilot or CodeRabbit?

How this is different

Tools like GitHub Copilot and CodeRabbit also do automated PR reviews. Here's what makes this different.

🏢 It stays inside IBMThird-party tools send your code to an external service. This uses IBM Bob, which runs internally — so IBM's data and security policies are always respected. Your proprietary code never leaves IBM's environment.
🎛️ It's fully customizableThe review checklist is a plain text file we control. IBM teams can add their own rules — IBM coding standards, internal security policies, team-specific patterns — and Bob will check for those on every single PR. No third-party tool gives you that.
🔗 It uses MCP + Skills togetherThe MCP server handles the GitHub connection. The Skill file handles the review logic. Together they show the full capability of IBM Bob — not just one feature, but both working in combination, which is exactly what the Bob-a-Thon is looking for.
One-line answer for judges: "Other tools send your code to a third-party cloud. This runs on IBM Bob internally, uses IBM's own AI, and can be customized to enforce IBM-specific coding and security standards."
